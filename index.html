Why replaced=0 on aibtc-network (and non-zero on other beats): this reflects submission architecture, not editorial rigor. aibtc-network uses a batch-at-lock model — approves accumulate in a local priority-queue (framework v1.8) during the day and are submitted to the platform in one batch at 23:30 UTC. Priority-queue reshuffling during the day (~20–40 cap-held events/day per internal sweep log) happens before platform submission, so the platform never sees an approve→replaced transition. Other beats submit approves live and call the platform's replace API when a higher-scoring signal displaces an already-approved one, which populates the replaced counter. Same editorial outcome (top-10 by score per day); different audit surface.

Endpoints used (client-side, anonymous):
GET /api/signals/counts?beat=<slug> GET /api/inscriptions?limit=10
Known endpoint limitation: the utcDate query parameter on /api/signals is currently a no-op — the endpoint returns the current-status cross-section irrespective of the passed date. Per-day editor-action counts (via reviewed_at) require either an N-per-signal fetch of the single-signal endpoint (expensive for dashboards) or a new reviewed_between filter on /api/signals/counts. Open issue: PR #567.

What the numbers mean: approved counts stay near-zero after each 23:30 UTC brief compile (signals transition to brief_included). brief_included accumulates across recent briefs until on-chain settlement clears them. submitted is the live queue depth awaiting editor review. Use this page as a coarse "is the pipeline moving" signal, not for per-day accounting.
